# Event Loop Agent - C/C++ 全量代码扫描优化计划

> 本优化计划仅聚焦于 C/C++ 代码扫描能力的提升，不涉及其他语言。

---

## 1. 当前 C/C++ 扫描能力评估

### 1.1 已有能力

| 能力项 | 现状 | 评价 |
|--------|------|------|
| 并发调度 | Python asyncio + Semaphore(3) + 独立 nga 进程 | 良好，事件循环模型稳定 |
| 输入模式 | Diff 模式 / 文件列表模式 | 满足基本场景 |
| 超时控制 | 动态超时(diff 行数) + SIGTERM/SIGKILL | 机制完善 |
| 规则覆盖 | 10 条规则(4条无线专用 + 6条通用低错) | 规则数量和深度均不足 |
| 扫描粒度 | 单文件独立扫描 | 缺少跨文件关联分析 |
| 输出格式 | Markdown 报告 + 运行日志 | 可读性好，但缺少结构化格式 |

### 1.2 核心短板

1. **100% 依赖外部 nga 进程**：无本地静态分析能力，每次扫描都是全量调用 LLM，成本高、延迟大
2. **单文件孤立扫描**：无法分析函数调用链、全局变量跨文件影响、头文件变更的级联影响
3. **规则深度有限**：10 条规则以"模式匹配"为主，缺少数据流分析、控制流分析、符号执行等深度能力
4. **增量扫描粗放**：仅按 diff 行数调整超时，未分析变更的实际影响范围

## 2. 优化方向（C/C++ 专用）

### 2.1 规则深度扩展（P0 - 高优先级）

#### 2.1.1 无线通信协议栈专用规则

| 规则编号 | 规则名称 | 严重等级 | 说明 |
|---------|---------|---------|------|
| RULE-011 | 定时器泄漏 | HIGH | `start_timer` 后缺少对应的 `stop_timer`，或异常路径未停止 |
| RULE-012 | 状态机完整性 | HIGH | RRC/MAC 状态转换 switch 是否覆盖所有枚举值，是否存在死状态 |
| RULE-013 | 整数溢出 | CRITICAL | `offset + length`、`size * count` 等运算是否可能溢出回绕 |
| RULE-014 | 字节序转换 | HIGH | 网络消息字段是否经过 `ntohs`/`ntohl` 转换，结构体强转时对齐问题 |
| RULE-015 | 并发竞争条件 | HIGH | 全局调度表/消息队列的访问是否受锁保护，`lock`/`unlock` 是否配对 |
| RULE-016 | 资源池耗尽 | HIGH | 内存池/消息池的分配是否检查剩余容量，释放后是否回收到正确池 |
| RULE-017 | 回调函数注册与注销 | MEDIUM | 事件回调注册后，模块销毁时是否注销，避免野指针回调 |
| RULE-018 | 消息序列号回绕 | CRITICAL | SN (Sequence Number) 递增后是否处理回绕到 0 的情况 |
| RULE-019 | 位域操作安全 | MEDIUM | 位域赋值是否越界，位掩码宏定义与实际位宽是否一致 |
| RULE-020 | 宏定义副作用 | MEDIUM | 带参数的宏是否用 `do { } while(0)` 包裹，参数是否被多次求值 |

#### 2.1.2 通用 C/C++ 安全规则补充

| 规则编号 | 规则名称 | 严重等级 | 说明 |
|---------|---------|---------|------|
| RULE-021 | 格式化字符串漏洞 | CRITICAL | `printf`/`sprintf`/`LOG_XXX` 中使用用户可控的格式字符串 |
| RULE-022 | 命令注入 | CRITICAL | `system()`/`popen()`/`exec()` 的参数是否经过严格过滤 |
| RULE-023 | 密码学误用 | HIGH | `rand()` 用于密钥、硬编码密钥/IV、ECB 模式、MD5/SHA1 用于安全场景 |
| RULE-024 | 不安全函数使用 | HIGH | `strcpy`/`sprintf`/`gets` 等无界函数，应替换为 `strncpy`/`snprintf` |
| RULE-025 | 返回栈地址 | CRITICAL | 函数返回局部变量的地址（`return &local_var`） |
| RULE-026 | 双重释放 | CRITICAL | 同一内存被 `free` 两次，或 `free` 后未置 NULL |
| RULE-027 | 有符号/无符号混用 | MEDIUM | `size_t` 与 `int` 混用导致负数被解释为大正数 |
| RULE-028 | 隐式类型转换 | MEDIUM | 窄类型赋值给宽类型时的精度丢失，`float` 与 `double` 混用 |
| RULE-029 | `const` 正确性 | LOW | 函数参数应为 `const` 却未标记，导致意外的修改风险 |
| RULE-030 | 异常安全(C++) | HIGH | C++ 构造函数中抛异常是否导致资源泄漏，析构函数是否 `noexcept` |

### 2.2 集成本地 SAST 引擎（P0 - 高优先级）

当前系统 100% 依赖外部 nga 进程（LLM），每次扫描都需要启动子进程并等待 LLM 推理，存在以下问题：
- **成本高**：LLM 调用费用高，全量扫描不可承受
- **延迟大**：每个文件等待数秒到数十秒
- **不稳定**：受网络、LLM 服务可用性影响

**建议方案：引入本地 SAST 工具作为"预处理层"**

```
文件输入
  │
  ├──▶ 本地 SAST 引擎（Cppcheck + Semgrep + Clang SA）
  │     ├── 快速定位可疑代码位置
  │     └── 输出结构化中间结果（SARIF）
  │
  └──▶ 结果聚合
        │
        ├── 低置信度 / 复杂逻辑 ──▶ 转发给 nga（LLM）深度分析
        │
        └── 高置信度 / 模式明确 ──▶ 直接输出，无需 LLM
```

**工具选型**：

| 工具 | 用途 | 与当前系统的集成方式 |
|------|------|-------------------|
| **Semgrep** | 自定义规则引擎，完美匹配 RULE-001~RULE-030 | 在 orchestrator 中直接调用 `semgrep --config=rules/` |
| **Cppcheck** | C/C++ 静态分析，擅长内存泄漏、越界、未初始化 | 作为补充引擎，处理 Semgrep 不擅长的分析 |
| **Clang Static Analyzer** | 深度路径敏感分析，符号执行 | 用于复杂控制流的深度分析 |
| **CodeQL** | GitHub 提供的语义分析引擎 | 可选，适合 GitHub 仓库 |

**预期收益**：
- 70% 以上的明确模式问题（RULE-001~RULE-010 的大部分）可由本地工具直接检出
- 仅将 LLM 用于需要语义理解的复杂场景（跨文件调用链、业务逻辑漏洞）
- 全量扫描时间从"小时级"降至"分钟级"

### 2.3 跨文件/跨模块分析（P1 - 中高优先级）

当前系统的最大架构限制：每个文件一个独立的 nga session，文件之间完全隔离。

对于 C/C++ 代码，以下场景必须跨文件分析：

| 场景 | 说明 | 示例 |
|------|------|------|
| 头文件变更级联 | 修改结构体定义，需检查所有引用该头文件的源文件 | `RrcMsgHdr` 增加字段后，所有解引用该结构体的文件需检查偏移 |
| 全局变量跨文件 | 全局变量在 A.c 定义，在 B.c 修改，在 C.c 读取 | 检查全局变量是否有并发保护 |
| 函数调用链 | 从入口函数追踪到最底层实现 | `handle_rrc_setup()` → `parse_msg()` → `tlv_decode()` |
| 宏定义影响 | 宏在头文件中定义，修改后影响所有展开位置 | `#define MAX_PDU_SIZE 1024` 改为 2048 后的缓冲区检查 |

**实现方案**：

1. **构建代码索引（Code Index）**
   - 使用 `clang` 的 `-Xclang -ast-dump` 或 `clangd` 的索引能力
   - 或者使用开源工具 `ctags`/`cscope` 的现代化替代 `global`/`clangd-index`
   - 建立：符号定义表、符号引用表、头文件包含图

2. **变更影响分析（Impact Analysis）**
   ```
   当文件 X 发生变更时：
     - 如果 X 是头文件：找到所有 #include X 的源文件，加入扫描队列
     - 如果 X 修改了函数实现：找到该函数的 caller 列表，加入扫描队列
     - 如果 X 修改了全局变量：找到所有读写该变量的文件，加入扫描队列
   ```

3. **调用链提取（Call Graph）**
   - 使用 `clang-check` 或 `bear` + `compiledb` 生成 compile_commands.json
   - 使用 `clang` 的 LibTooling 提取调用图
   - 扫描时，将变更函数及其上下游调用链打包为一个"分析单元"发给 nga

### 2.4 增量扫描智能化（P1 - 中高优先级）

当前 Diff 模式的逻辑：
1. `git diff --name-only` 获取变更文件
2. 对每个文件启动一个 nga session
3. 超时按 diff 行数线性增长

**优化方向**：

#### 2.4.1 语义级增量（不只是看变更行）

| 变更类型 | 影响范围 | 扫描策略 |
|---------|---------|---------|
| 日志修改（`LOG_INFO` → `LOG_DEBUG`） | 仅当前文件 | 快速扫描，低超时 |
| 指针偏移调整（`p += len` 改为 `p += length`） | 当前文件 + 调用者 | 中等深度扫描 |
| 结构体字段增删 | 所有包含该头文件的源文件 | 深度扫描，关联分析 |
| 全局锁策略变更 | 所有访问该锁的文件 | 全量关联扫描 |
| 协议版本号升级 | 整个协议栈模块 | 全量深度扫描 |

**实现方式**：在 orchestrator 中增加一个 "Change Classifier" 模块，用正则/AST 快速分类变更类型，然后选择扫描策略。

#### 2.4.2 增量缓存

- 为每个文件维护一个"规则命中缓存"（文件哈希 → 规则命中列表）
- 文件未变更时直接复用上一次结果
- 头文件变更时，仅使引用该头文件的文件缓存失效

### 2.5 报告与度量体系（P2 - 中优先级）

当前输出：Markdown 报告 + 日志文件，适合人工阅读，不适合系统集成。

**优化内容**：

1. **SARIF 格式输出**
   - SARIF (Static Analysis Results Interchange Format) 是行业标准化格式
   - 兼容 GitHub Advanced Security、GitLab SAST、Azure DevOps 等 CI/CD 平台
   - 支持富文本代码片段、位置信息、修复建议、规则元数据

2. **缺陷趋势追踪**
   - 按 commit / tag 维度记录缺陷数量
   - 追踪：新增缺陷数、修复缺陷数、遗留缺陷数、误报率
   - 生成趋势图（可用 Grafana / 简单 HTML 报告）

3. **误报反馈闭环**
   - 支持开发者在报告中标记 "误报" / "确认"
   - 误报数据反馈给规则引擎，用于调整置信度阈值
   - 对 nga 的结果：可通过后续 prompt 告知 "上次分析 X 行是误报"

### 2.6 CI/CD 集成（P2 - 中优先级）

**目标**：将扫描从"本地脚本"升级为"持续集成门禁"

1. **GitLab CI 集成**
   ```yaml
   # .gitlab-ci.yml
   code_scan:
     stage: test
     script:
       - python orchestrator.py --diff $CI_MERGE_REQUEST_DIFF_BASE_SHA --repo . --timeout 600
     artifacts:
       reports:
         sast: reports/sast-report.sarif
     rules:
       - if: $CI_MERGE_REQUEST_ID
   ```

2. **GitHub Actions 集成**
   ```yaml
   # .github/workflows/code-scan.yml
   - name: Run C/C++ Security Scan
     run: python orchestrator.py --diff ${{ github.event.pull_request.base.sha }} --repo .
   ```

3. **MR 门禁机制**
   - 扫描发现 CRITICAL 级别问题 → 阻塞 MR 合并
   - 扫描发现 HIGH 级别问题 → 需要安全团队审批
   - 扫描发现 MEDIUM/LOW → 建议修复但不阻塞

4. **扫描结果持久化**
   - 将结果写入数据库（SQLite/PostgreSQL）
   - 提供 Web 界面查看历史扫描结果、趋势、统计

## 4. 技术选型建议

### 4.1 本地 SAST 引擎选型

| 工具 | 适用场景 | 是否推荐 | 理由 |
|------|---------|---------|------|
| **Semgrep** | 自定义规则、快速模式匹配 | 强烈推荐 | 规则易写（YAML），社区规则丰富，对 C/C++ 支持良好 |
| **Cppcheck** | 通用 C/C++ 静态分析 | 推荐 | 误报率低，擅长内存问题，开源免费 |
| **Clang Static Analyzer** | 深度路径敏感分析 | 推荐 | 与 clang 编译器集成，路径覆盖率高 |
| **Infer** (Facebook) | 内存泄漏、空指针 | 可选 | 对 C/C++ 支持一般，更擅长 Java/ObjC |
| **CodeQL** | 语义分析 | 可选 | 功能强大但规则编写门槛高，适合 GitHub 生态 |

### 4.2 跨文件分析工具选型

| 工具 | 用途 | 是否推荐 |
|------|------|---------|
| **clang-check + LibTooling** | AST 提取、符号索引 | 强烈推荐 |
| **compiledb** / **bear** | 生成 compile_commands.json | 强烈推荐 |
| **clangd** | LSP 索引、跳转定义 | 可选（可作为索引后端） |

### 4.3 输出格式选型

| 格式 | 用途 | 是否推荐 |
|------|------|---------|
| **SARIF 2.1.0** | CI/CD 集成、标准化 | 强烈推荐 |
| **Markdown** | 人工阅读、MR 评论 | 保留（当前已有） |
| **JSON** | 程序消费、数据库存储 | 保留（内部使用） |
