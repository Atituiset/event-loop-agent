# 低错问题
无线通信系统中存在很多高低版型消息的兼容性和适配性问题
这是一个极其痛点且极具行业特征的问题。在无线通信系统（如 4G/5G 的 RRC、MAC、NAS 层协议）的演进过程中，协议版本的迭代会产生海量的高低版型消息。

在数百万行代码的庞大系统中，高版本基站与低版本终端（或反之）通信时，如果 C/C++ 代码在消息解包和适配层处理不当，极易引发越界访问、静默丢包甚至系统宕机。

这种兼容性问题在代码层面通常表现为以下几种典型的“雷区”，我们可以专门针对这些雷区为 OpenCode Agent 编写定向扫描规则。

### 1. TLV 解析陷阱与向前兼容性

为了实现向前兼容，无线消息通常大量采用 **TLV (Type-Length-Value)** 或嵌套 TLV 结构。高版本增加的新字段会被封装在新的 Type 中，低版本系统理应通过 Length 字段直接跳过未知扩展。
* **常见代码缺陷：** 解析循环中没有对 `Length` 的合法性做严格校验，或者在遇到未知的 `Type` 时，直接 `break` 中断了解析，而不是偏移指针跳过该字段，导致后续合法字段丢失。
* **Agent 扫描策略：** 引导 Agent 专门审查提取 TLV 结构中 `Length` 字段的逻辑，要求其验证在指针执行 `p += length` 之前，是否对剩余 Buffer 总长度进行了越界检查（Boundary Check）。

1. 针对手写 TLV 解析的“边界防御”提示词
此提示词专注于无线通信中最致命的指针越界问题。

角色： 你是一名资深嵌入式通信协议栈架构师。
任务： 审查这段 TLV (Type-Length-Value) 消息解析代码。
重点： > 1. 指针溢出风险： 检查每次执行 p_buf += length 或类似操作前，是否严格校验了 remaining_len？
2. 向前兼容性： 如果收到当前版本未定义的 Type，代码是否能利用 Length 字段安全跳过该字段并继续解析后续字段，还是会直接报错/死循环？
3. 长度字段可信度： 检查 Length 字段本身是否可能为极大值导致指针加法回绕（Wrap-around）。
输出要求： 仅报告确实存在越界风险或导致解析挂死的逻辑漏洞，忽略已包含防御宏的行。

### 2. 结构体强转与内存越界
早期或底层的模块为了追求极致性能，有时会绕过序列化框架，直接将一段内存 Buffer 强制转换为 C 结构体指针进行访问。
* **常见代码缺陷：** 当高版本协议在消息尾部追加了新字段，而低版本的接收端依然使用旧的、较短的结构体去映射内存时，如果后续逻辑尝试读取新增字段，就会发生内存越界。
* **Agent 扫描策略：** 重点排查包含 `memcpy`、`reinterpret_cast` 或 C 风格强转的网络包处理函数。让 Agent 检查转换前是否对比了实际接收到的 `Payload Size` 与 `sizeof(TargetStruct)`。

### 3. Switch-Case 的“黑洞”
在状态机或消息分发中心，通常有巨型的 `switch-case` 来处理不同的 Message ID。
* **常见代码缺陷：** 缺少防御性的 `default` 分支，或者 `default` 分支中没有正确的丢弃/错误上报逻辑。当收到高版本的新消息时，程序可能直接陷入未定义行为。
* **Agent 扫描策略：** 通过抽象语法树（AST）辅助 Agent 提取所有的消息分发 `switch` 块，强制校验是否所有未显式处理的 `Case` 都安全回退。

### 4.针对 ASN.1 生成代码的“逻辑空洞”提示词
此提示词专注于开发人员在调用 ASN.1 库时容易忽略的 Optional 字段检查。

角色： 你是一名 5G 物理层协议开发专家。
任务： 审计这段基于 ASN.1 生成结构体编写的业务代码。
场景： 考虑高版本基站下发了包含可选字段（Optional Fields）的消息，而当前低版本代码正在运行。
重点：

空指针解引用： 检查在访问任何标记为 OPTIONAL 的结构体成员（如 p_struct->ext_v15）前，是否显式判断了其 Presence 标志位或是否为 NULL？

枚举适配： 检查针对 CHOICE 或 ENUM 类型的 switch 语句，是否包含能处理未来版本新增定义的 default 分支？
输出要求： 列出所有潜在的未检查访问点，并给出防御性代码建议。
